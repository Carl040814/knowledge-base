# [密码学（Cryptography）](https://aiweb3.school/zh/handbook/web3/cryptography)

> 约 6 分钟阅读 · 2026-05-12

**核心原则**：Web3 里的"账户""签名""地址""所有权"都建立在密码学之上。你不需要成为密码学研究者，但必须知道哪些动作是在证明身份，哪些是在授权资产，哪些东西一旦丢失就无法恢复。

## 第一性原理

链上身份不是由平台发给你的，而是由你能否证明自己控制某个私钥决定的。

Web3 的账户系统把"谁有权限"从中心化数据库转成了可验证的数学关系。好处是开放、可组合、无需平台许可；代价是用户和应用都要承担更明确的密钥与签名风险。

- **私钥是控制权**：丢失私钥意味着失去控制权，泄漏私钥意味着别人获得控制权。
- **签名是授权证据**：签名内容必须能被人读懂，不能只展示一串难懂数据。
- **哈希是承诺**：哈希不能还原原文，但可以用来验证数据有没有被改过。

## 知识节点

| 节点 | 解释 |
|------|------|
| **Hash（哈希）** | 把任意数据变成固定长度指纹，验证数据是否一致。输入差一个字符，输出完全不同。链上用途：交易哈希、区块哈希、合约字节码哈希、数据承诺。哈希不是加密，不能"解密回原文" |
| **Public Key（公钥）** | 与私钥成对，可以公开。用来推导地址或验证签名。不能从公钥反推私钥。地址是从公钥进一步处理得到的短标识 |
| **Private Key（私钥）** | 账户控制权本身。用来生成签名，证明你控制某个账户。私钥一旦泄漏，攻击者可直接转走资产。**不截图、不上传云盘、不粘贴给网页、不发给 AI、不进代码仓库** |
| **Signature（签名）** | 对某段具体消息或交易内容的授权证明。私钥生成，公钥/地址验证。最容易出问题：用户看不懂自己签了什么。产品应展示签名目的、域名、链 ID、合约地址、额度、过期时间 |
| **Merkle Tree** | 用一组哈希把大量数据组织起来，只拿到少量证明也能验证某条数据属于整体。常见于空投名单、轻客户端、状态证明 |

## 安全原则

- 地址 ≠ 匿名（链上公开可查）
- 地址不是身份本身，只是可验证控制关系的标识
- "看起来像官方地址"没有意义——必须和合约文档、官网、区块浏览器交叉验证
- 私钥管理：不截图、不上传、不粘贴、不发给 AI、不用主钱包测试陌生 dApp

## 在 AI × Web3 中的位置

AI Agent 如果要解释交易、判断授权、生成链上操作，必须理解签名和密钥的边界。模型可以解释"这笔授权在批准某个 token 额度"，但不能替用户保管私钥，也不能在用户不理解内容时推动签名。

AI 输出本身也可能需要被签名、哈希或记录，作为后续审计证据。

## 最小实践

1. 创建测试钱包，不放真实资产
2. 在测试环境发起普通转账或消息签名
3. 在区块浏览器观察地址、交易哈希、签名提示和链 ID
4. 写下哪些信息可公开、哪些绝不能公开
5. 比较"签名消息"和"发送交易"的区别
