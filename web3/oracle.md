# [预言机（Oracle）](https://aiweb3.school/zh/handbook/web3/oracle)

> 约 5 分钟阅读 · 2026-05-12

**核心原则**：区块链自己不能天然知道链外世界发生了什么。Oracle 的作用，是把价格、数据、事件结果等外部信息，以可被合约使用的方式带进链上。

预言机不是"真实世界 API"，而是链上合约愿意信任的一套数据提交和验证机制。

## 第一性原理

合约只能按链上可见的数据执行，所以外部数据一旦进入链上，就会变成协议规则的一部分。

如果一个借贷协议使用某个价格 feed 判断清算，那么这个 feed 的更新频率、数据源、聚合方式和异常处理，都会影响用户资产安全。

- **数据源是信任边界**：谁提供数据、谁聚合、谁能更新，都必须清楚
- **延迟会变成风险**：价格变化很快时，旧数据可能导致错误清算或坏账
- **异常要有保护**：合约应该处理 stale price、极端跳变和 feed 暂停

## 知识节点

| 节点 | 解释 |
|------|------|
| **Price Feed** | 最常见形式，为合约提供资产价格。读取时检查：feed 对应资产对、decimals、更新时间是否过旧、返回值是否异常、feed 地址是否正确 |
| **Data Feed** | 不只价格，还包括储备证明、利率、宏观数据、体育结果等。只要合约依赖链外数据，就要问：数据从哪来、如何更新、谁能提交、是否可验证、出错怎么处理 |
| **Oracle Risk** | 链外数据进入链上执行时的系统性风险：数据源被操纵、feed 更新延迟、聚合节点离线、decimals 理解错误、低流动性资产价格被攻击、未检查 stale price、升级/权限变更不透明 |
| **AI Oracle** | 把模型推理/评分/分类结果提交链上。比价格 feed 更复杂：涉及模型版本、输入数据、提示词、评估标准、争议处理。设计时需考虑输入可追溯、输出可复核、允许挑战/仲裁 |

## 在 AI × Web3 中的位置

Oracle 是 AI x Web3 的关键桥梁。AI Agent 需要读取链上链外数据；合约使用 AI 结果需把模型输出转成可验证数据。

不要把 AI Oracle 想成"模型说什么合约就执行什么"。更合理：AI 给出结果，记录来源和置信度，高风险引入 human-in-the-loop、挑战期或经济惩罚。

## 最小实践

1. 找一个链上 ETH/USD price feed
2. 查看 feed 地址、decimals、最新价格和更新时间
3. 思考价格延迟 30 分钟哪些协议动作会受影响
4. 写出合约读取此 feed 应检查的条件
5. 设计一个 AI Oracle 输出场景，列出必须记录的输入、模型版本和争议流程
