# [开发栈（Dev Stack）](https://aiweb3.school/zh/handbook/web3/dev-stack)

> 约 9 分钟阅读 · 2026-05-12

**核心原则**：Web3 开发栈不是一组随机工具名，而是一条从写合约、测合约、部署合约、连接钱包、调用合约到监控结果的工程链路。工具选型的目标是让链上开发更可验证、更可复现、更少事故。

## 第一性原理

Web3 工具链的核心是把不可逆执行前移到可测试、可模拟、可审查的流程里。

链上交易一旦成功，后悔成本很高。因此开发工具要尽量把错误暴露在本地、测试网、模拟环境和 code review 阶段，而不是等到主网上线后再发现。

- **本地优先复现**：合约逻辑、部署脚本和前端调用都应能在本地或测试网跑通
- **地址和 ABI 要版本化**：前端调用的到底是哪份合约，必须能追踪
- **安全库不是免审计**：OpenZeppelin 等库降低基础风险，但组合逻辑仍要自己验证

## 知识节点

### 最小开发链路（六步）

1. 在本地或浏览器 IDE 写出合约
2. 编译合约，得到 bytecode 和 ABI
3. 在本地链或测试网部署合约
4. 写测试覆盖核心状态变化和权限边界
5. 前端用合约地址和 ABI 读取或发送交易
6. 在区块浏览器验证源码、交易和事件

### 工具链

| 工具 | 定位 | 说明 |
|------|------|------|
| **Remix** | 浏览器 Solidity IDE | 适合快速理解 Solidity、部署小合约。低门槛，不适合长期替代工程化 repo |
| **Hardhat** | JS/TS 合约工程框架 | 本地网络、编译、测试、部署脚本、插件生态。适合前端团队 |
| **Foundry** | CLI + Solidity-native 测试 | forge、cast、anvil。编译快、测试快，适合高强度合约开发 |
| **OpenZeppelin** | 合约标准库 | ERC-20/721、AccessControl、Ownable 等。不能替代业务逻辑审查 |
| **viem / wagmi** | TypeScript 链交互 | viem 偏底层类型安全，wagmi 偏 React 钱包/账户 hooks |

### 前端四种状态（必须分开处理）

1. 钱包是否已连接
2. 用户当前在哪条链
3. 合约读取结果是否正在加载或已过期
4. 写交易处于：等待签名 / 已广播 / 等待确认 / 成功 / 失败

## 在 AI × Web3 中的位置

AI 可以显著提升 Web3 开发效率：解释 ABI、生成部署脚本、补测试、排查交易失败。但 AI 参与开发栈后，验证流程要更明确。高风险命令需要人工确认。

## 最小实践

1. 用 Remix 部署一个极简计数器合约
2. 用 Hardhat 或 Foundry 建本地工程，写测试
3. 记录合约地址、ABI、部署网络、交易哈希
4. 用 viem 或 wagmi 从前端读取 + 发起交易
5. 在区块浏览器确认 event 被正确发出
6. 写下哪些信息进版本控制，哪些放 .env
