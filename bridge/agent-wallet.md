# [Agent 钱包（Agent Wallet）](https://aiweb3.school/zh/handbook/bridge/agent-wallet)

> 约 12 分钟阅读 · 2026-05-12

**核心原则**：Agent Wallet 不是"给 AI 一个钱包私钥"——真正的问题是 Agent 能做什么链上动作、有没有额度/时间/对象/风险限制、用户什么时候必须重新确认。

它不是让 AI 接管资产，而是让用户把一小部分、可撤销、可审计的权限交给 Agent。

## 第一性原理

控制权不能交给一个概率系统。Agent 只能拿到可验证、可限制、可撤销的行动空间。

Agent 的判断来自模型、上下文和工具返回值——可能有用，也可能被错误信息或恶意文档带偏。所以 Agent Wallet 的关键不是让 Agent 更像"真人用户"，而是把它的行动范围变成可检查的规则。

- **Agent 不直接控制主资产**：模型可生成计划，但不能拿用户主私钥。高风险动作必须交回人确认
- **钱包不只是签名按钮，而是权限系统**：表达时间、金额、合约、方法、资产、接收方和撤销条件
- **自动化必须绑定撤销能力**：用户随时能看到 Agent 拿了什么权限、做过什么、还能做什么、在哪关掉

## 知识节点

| 节点 | 说明 |
|------|------|
| **AA Wallet** | 基于 Account Abstraction 的钱包。关键不是"更高级"，而是账户能表达规则：谁能操作、能操作什么、什么时候失效、超出边界怎么办 |
| **Smart Account** | Agent Wallet 的执行边界。定义：谁可发起操作、哪些需确认、哪些合约可调、每天限额、权限自动失效时间、异常暂停机制 |
| **Safe** | 团队/DAO 多签基础设施。天然适合把执行权拆开：Agent 生成草稿或触发低风险模块，转账/升级/大额仍需多签确认 |
| **Session Key** | 临时有限权限小钥匙。约束：有效时间、可调合约/方法、单笔金额和总额、可操作 token、是否允许转出外部地址、是否可随时撤销。**Session Key 不是小号私钥，是一组受限能力** |
| **Policy** | 系统可检查的执行规则：每天最多 X USDC、只能白名单合约、只能 swap 不能无限 approve、滑点 >1% 停止、NFT 转出必须人工确认 |
| **Guard** | 确定性拦截层，拒绝不符合 policy 的交易。分工：Agent 生成候选动作，Guard 拒绝越界动作 |
| **Simulation** | 签名前预演交易结果：你会支付多少、会收到什么、哪些授权会改变、哪个合约被调用、失败损失什么。不只是技术检查，是用户理解和确认的入口 |
| **Revocation** | 权限撤销机制。不只用户主动关——系统也应自动收紧：session key 到期、额度用完、多次失败、检测到异常地址、行为偏离、用户长时间未确认 |
| **Human Check** | 分层确认：低风险自动、中等先模拟再确认、高风险明确展示影响+确认、超 policy 直接拒绝。让用户看懂在批准什么 |

## 一条完整执行链路

```
用户给出目标和约束
    ↓
Agent 读取上下文并生成计划
    ↓
系统把计划转成受限交易或操作
    ↓
Guard + Simulation 检查风险
    ↓
用户确认关键动作  ←── 高风险必须到这步
    ↓
Smart Account / Safe 执行
    ↓
日志记录 Agent 做过什么
```

最容易出问题的是第 3→6 步——很多项目把"模型生成计划"做得很顺，但没有认真处理交易模拟、权限校验、撤销入口和审计记录。

## 在 AI × Web3 中的位置

Agent Wallet 是 AI 和 Web3 的连接点。AI 负责理解目标 + 生成计划，Web3 钱包负责身份 + 资产 + 签名 + 执行。Agent Wallet 把两边接起来，但不能让模型越过用户控制权。

## 最小实践

设计"Agent 受限支付钱包"：用户允许 Agent 一天内最多花 5 USDC 支付白名单 API。写出：

1. 用户授权的额度、时间和目标地址
2. Agent 可自动执行的操作
3. 哪些操作必须人工确认
4. 交易发出前如何模拟和展示结果
5. 超出 policy 时系统如何拒绝
6. 用户在哪查看和撤销权限
7. 每次支付如何留下审计记录

四个场景：正常支付 ✅ / 超限被拒 ❌ / 目标不在白名单被 Guard 拦 ❌ / 用户撤销 session key，Agent 失去执行能力 ❌
