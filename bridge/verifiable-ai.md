# [可验证 AI（Verifiable AI）](https://aiweb3.school/zh/handbook/bridge/verifiable-ai)

> 约 6 分钟阅读 · 2026-05-12

**核心原则**：Verifiable AI 解决的不是"AI 对不对"，而是"AI 输出来自哪个模型、什么输入、什么环境、是否被篡改"。

## 第一性原理

在信任模型之前，先验证模型的运行环境、输入和输出。

- **模型和输入不能被事后伪造**：必须有签名或证明固定执行条件
- **验证不等于信任**：TEE/ZK 证明模型确实跑了，但不能证明模型一定给出最佳答案
- **验证粒度要有取舍**：全量推理验证代价太大，关键步骤优先

## 知识节点

| 节点 | 说明 |
|------|------|
| **TEE** | 硬件可信执行环境，证明代码在隔离环境运行。证明：哪个模型、什么输入、输出是什么。局限：依赖硬件厂商、可能侧信道攻击、不证明模型正确性 |
| **ZK Proof** | 零知识证明验证计算正确性。可用性随 ZK 技术演进提升。现实：复杂模型证明开销大，适合关键步骤或小模型 |
| **Optimistic Verification** | 乐观执行 + 挑战期：先信任结果，设窗口期允许质疑。适合非实时、可等待验证、有经济惩罚的场景 |
| **Committee / Multi-prover** | 多验证者/多模型交叉验证。不同环境跑同一输入，结果不一致触发仲裁。代价是成本和延迟 |
| **Attestation** | 简单证明形式：谁、什么时候、用什么模型、输入 hash、输出 hash。不证明正确性但提供追溯基础 |
| **Verifiable Compute** | 通用可验证计算（RISC Zero 等 zkVM）。把整个推理过程放入可验证虚拟机的长期方向 |

## 在 AI × Web3 中的位置

Verifiable AI 是 Agent 执行的可信基础。Agent 生成交易建议→Attestation 记录输入/模型/版本→高风险场景加 TEE/ZK 证明→Optimistic 窗口允许挑战→链上只存 hash 和验证结果。

## 最小实践

设计 Agent 输出验证链：Agent 建议→Attestation（模型+输入+输出 hash）→高风险加 TEE 证明→链上存证明 hash→争议期→执行。
