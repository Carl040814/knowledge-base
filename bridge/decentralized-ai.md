# [去中心化 AI（Decentralized AI）](https://aiweb3.school/zh/handbook/bridge/decentralized-ai)

> 约 10 分钟阅读 · 2026-05-12

**核心原则**：去中心化 AI 不是"把模型放链上"，而是重新设计数据、模型、算力、推理、评测和收益分配的组织方式。

## 第一性原理

AI 系统真正需要去中心化的，不一定是模型本身，而是关键资源和关键决策的控制权。

- **先分层再判断**：数据/模型/算力/推理/评测/结算/治理不能混成一个词
- **开放性需要工程代价**：分布式网络引入延迟、质量波动、女巫攻击、节点作恶和治理低效
- **链上更适合做协调层**：记录权属、激励贡献、结算任务，比直接承载大规模模型训练更现实

## 知识节点

| 节点 | 说明 |
|------|------|
| **Model Market** | 模型能力被发现/比较/调用/付费的市场。关键：可替换性→模型选择从硬编码变路由决策 |
| **Data Market** | 数据授权/定价/验证/复用。现实方案围绕：可验证来源数据集、带授权条款数据、可复现 pipeline、贡献记录、标注样本 |
| **Compute Market** | GPU/CPU/存储/推理的开放算力市场。难点：节点不稳定、延迟波动、可复现性、结果验证、隐私。适合先从批处理/非关键做 |
| **Inference Network** | 推理节点网络：请求→选节点→执行→返回→计费/审计。需处理节点发现、模型版本、路由、限流、失败、验证、付款 |
| **Model Routing** | 多模型/节点间调度选择。按：任务类型、风险等级、数据敏感度、成本预算、延迟要求、评测反馈 |
| **Quality Benchmark** | 开放网络质量保证。不只测模型分数，还要测：结果稳定性、节点执行一致性、延迟/失败率、可复现性、刷分检测 |
| **Settlement** | 从"开放调用"到"可持续网络"的关键。任务报价、请求参数、输出哈希、成功/失败/争议状态机、付款/托管/退款规则 |

## 在 AI × Web3 中的位置

去中心化 AI 是最底层基础设施议题。判断项目时固定问：它去中心化的是哪一层？链上负责什么？相比中心化改善了什么？开放网络的代价是否值得？

## 最小实践

设计开放推理网络规格：支持任务类型、角色（请求方/推理节点/评测者/支付方/仲裁）、请求字段、报价/返回/验证机制、验收/退款/争议处理、哪些上链哪些存 hash。
