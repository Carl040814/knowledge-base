# [AI 预言机（AI Oracle）](https://aiweb3.school/zh/handbook/bridge/ai-oracle)

> 约 7 分钟阅读 · 2026-05-12

**核心原则**：AI Oracle 是把模型输出提交给链上系统的机制。关键不是"如何上链"，而是如何记录输入、模型、版本、证明和争议。

## 第一性原理

当 AI 输出会影响链上资产或权限时，输出本身必须有来源、边界和争议机制。系统不能只记录"模型说通过"。

- **输入要可追溯**：模型看到了什么，不能只留最终答案
- **结果要结构化**：链上只消费明确字段（accepted: true、riskScore: 72），长文本存链下用 hash 关联
- **争议要提前设计**：谁能挑战、挑战期多久、如何复核

## 知识节点

| 节点 | 说明 |
|------|------|
| **AI Output** | 分"机器字段"和"人类解释"两层。机器字段进合约/规则，人类解释进 UI/报告。含 confidence、模型版本、输入 hash、输出 hash、生成时间 |
| **Data Feed** | 持续 AI 处理后的数据（地址风险评分、内容审核标签、交易意图分类）。需处理更新时间、版本变更、异常值、历史回放。比价格 feed 更易漂移 |
| **TEE** | Trusted Execution Environment：硬件级可信执行。可证明用了哪个模型、输入、版本。不解决模型本身好坏，但解决"结果是否真实来自声称的模型和输入" |
| **ZKML** | Zero-Knowledge ML：用零知识证明验证推理过程。理想状态可验证模型推理而不暴露输入。现实限制：计算开销大，适合小模型/关键步骤 |
| **Human Review** | 高风险判断放入类复核。TEE/ZK 给出"确实跑了这个模型"，但模型本身可能有偏见。需要人判断 + 争议机制 |
| **Dispute** | 挑战 AI Oracle 结果的机制。挑战期、挑战条件、仲裁方式、错误结果的经济后果必须提前定义 |

## 在 AI × Web3 中的位置

AI Oracle 连接模型输出和链上执行。越靠近资产决策，越需要多层验证：结构化输出→来源记录→可验证执行→争议机制。

## 最小实践

设计一个 AI Oracle 判断流程：任务→模型输入→结构化输出→TEE/ZK 证明→链上记录→争议期→最终执行。
